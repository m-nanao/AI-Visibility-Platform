# 10. AI協調開発フロー

このプロジェクトは、ユーザー・ChatGPT・Claude Codeの3者が役割分担して開発を進める。現在は**半自動フロー**（人間がタスクをClaude Codeへ手渡しする）で運用しており、将来的にはGitHub Issue / PR / CI / AIレビューを介した**完全自動フロー**への移行を目指す。

このドキュメントは、その運用ルールを記録する。タスクの依頼・実装・報告のフォーマットは [task_template.md](./task_template.md)、レビューのフォーマットは [review_template.md](./review_template.md) を参照。現状のプロダクト状態は [development_status.md](./development_status.md) を参照。

## 1. 役割分担

| 役割 | 主な責務 |
| --- | --- |
| **ユーザー** | 最終判断・承認。ChatGPTが作ったタスクをClaude Codeへ渡す。Claude Codeの報告を確認し、mainへのマージ・本番公開・環境変数変更等、[4. 人間承認が必須のこと](#4-人間承認が必須のこと)に該当する操作を承認する |
| **ChatGPT** | 要件整理・設計・タスク作成（[task_template.md](./task_template.md)形式）・Claude Codeの実装結果のレビュー（[review_template.md](./review_template.md)形式） |
| **Claude Code** | 渡されたタスクの実装・テスト・コミット・結果報告（[task_template.md](./task_template.md)の「作業後の報告形式」に従う） |

それぞれの持ち場を越えない。例えばClaude Codeはタスクに書かれていない設計判断や要件変更を自分で決めない（[3. AIに任せてよいこと](#3-aiに任せてよいこと)参照）。ChatGPTは実装そのものを行わず、レビューと次のタスク作成に専念する。

## 2. 半自動フロー（現状の運用）

```
1. ユーザーがChatGPTと会話し、要件・設計・レビュー指示を固める
2. ChatGPTがタスクを作る（task_template.md形式）
3. ユーザーがそのタスクをClaude Codeへ渡す
4. Claude Codeが実装・テストを行い、コミットする
5. Claude Codeが検証結果とコミット内容を報告する（task_template.mdの報告形式）
6. ユーザーがChatGPTへ実装結果・報告内容を共有する
7. ChatGPTがレビューする（review_template.md形式）
8. レビュー結果がPass以外の場合、ChatGPTが修正タスクを作り、3に戻る
9. Passになったら次のタスクに進む、または一区切りとする
```

このプロジェクトの現時点の状態（2026-07時点）はこのフローで運用されている。**CI（lint/test/buildの自動実行）のみ`.github/workflows/ci.yml`として導入済み**。GitHub Issue起点のタスク管理、Claude CodeによるPR作成、AIレビューの自動化、人間承認後の自動マージは、いずれもまだ導入していない（詳細は次章の表を参照）。

## 3. 将来の完全自動フロー（未着手・移行目標）

```
1. GitHub Issueとしてタスクが起票される（ChatGPTまたは人間が作成）
2. Claude Codeがそのissueを読み、ブランチを切って実装する
3. Claude CodeがPRを作成する
4. CI（.github/workflows/ci.yml）がlint/test/buildを自動実行する
5. AIレビュー（ChatGPT等）がPRの内容を確認しコメントする
6. 人間がレビュー結果とCI結果を確認し、問題なければ承認してマージする
```

現時点でこのプロジェクトにあるもの・ないものを整理すると以下の通り。

| 要素 | 状態 |
| --- | --- |
| GitHub Issue起点のタスク管理 | 未導入。タスクは`task_template.md`をユーザーが直接Claude Codeに渡す形 |
| Claude CodeによるPR作成 | 未導入。現状はmainブランチに直接コミットしている運用実績があるが、[4. 人間承認が必須のこと](#4-人間承認が必須のこと)にある通り、今後は最低限「1タスク1ブランチ」を原則とする |
| CI（lint/test/build自動実行） | `.github/workflows/ci.yml`として最小構成を追加済み（frontend: lint/test/build、backend: pytest）。PRトリガーでの自動レビュー連携はまだない |
| AIレビューの自動化 | 未導入。現状はユーザーが人手でChatGPTとClaude Codeの結果をやり取りする |
| 人間承認後の自動マージ | 未導入 |

完全自動フローへの移行は、まず「Claude CodeがPRを作る」「CIが自動で回る」の2点から段階的に進める想定。移行時期・優先順位は [05_tasks.md](./05_tasks.md) で管理する。

## 4. 人間承認が必須のこと

以下は、ChatGPT・Claude Codeがどれだけ確信を持っていても、**必ずユーザーの明示的な承認を経てから実行する**。

- **mainブランチへのマージ**（直接コミットではなくブランチ運用に移行した後）
- **ステージング環境（現行のVercel/Render）へのデプロイ操作、および将来の本番公開**（ドメイン変更、公開停止等を含む。現行のVercel/Renderは依頼者確認用ステージング環境であり正式な本番環境ではないが、デプロイ・停止・ドメイン変更はいずれも人間承認を経る。詳細は[09_deployment.md](./09_deployment.md)の「環境の位置づけ」参照）
- **DBマイグレーション**（Phase 5でPostgreSQLを導入した後の破壊的スキーマ変更全般）
- **APIキー・環境変数の追加・変更**（`PYTHON_ANALYSIS_API_URL`等の値変更、新しい秘密情報の導入）
- **外部API課金が増える可能性のある変更**（有料プランへのアップグレード、従量課金APIの呼び出し追加等）
- **認証・権限・セキュリティ関連の変更**（CORS設定変更、認証機能の追加、アクセス制御の変更等）

これらは [CLAUDE.md](../CLAUDE.md) にも同じ内容を記載し、Claude Codeが作業開始前に必ず参照する運用とする。

## 5. AIに任せてよいこと

- タスクに明記された範囲内での実装・リファクタ・バグ修正
- タスクに明記された検証コマンド（lint/test/build/pytest）の実行と結果報告
- タスクの完了に伴う関連docsの更新（実装と矛盾しないようにするため）
- 軽微な表記ゆれ・typo・コメントの改善（タスクの主目的を妨げない範囲）
- ローカルの開発用プロセス（devサーバー等）の起動・停止。ただし**ユーザーが起動した既存のプロセスを、ユーザーの許可なく停止しない**（[CLAUDE.md](../CLAUDE.md)参照）

## 6. 1タスクの粒度

- 1つのタスクは、**1つの機能追加・1つの修正・1つの整理作業**に閉じるようにする。「ついでに直す」を避ける（[task_template.md](./task_template.md)の「対象外」欄で明示する）。
- 実装ファイル数の目安ではなく、「レビューする側が1回で読める・判断できる大きさか」を基準にする。
- [05_tasks.md](./05_tasks.md)に記載された項目のうち、粒度が大きすぎるもの（複数の設計判断や複数コンポーネントにまたがるもの）には「※粒度大」の注記を付けている。着手前にChatGPT側で1件ずつのタスクへ分解してからClaude Codeへ渡す。

## 7. タスク失敗時の扱い

- Claude Codeが完了条件を満たせなかった場合、無理に「完了」として報告しない。[task_template.md](./task_template.md)の報告形式にある「Remaining Issues」に、何が・なぜできなかったかを具体的に書く。
- 検証コマンド（lint/test/build/pytest）が1つでも失敗した状態で「完了」と報告しない。失敗したまま報告する場合は、その理由（環境起因か、実装起因か）を明記する。
- 仕様・設計が不明瞭でタスクの記述だけでは判断できない場合、大規模な推測実装をせず、いったん立ち止まってユーザー（またはタスクを作ったChatGPT）に確認する。

## 8. 修正ループの上限

- 同一タスクに対する「レビュー→修正」のループは、目安として**3回まで**とする。3回を超えて収束しない場合は、タスクの前提（要件・設計）自体を見直すべきサインとみなし、ChatGPTとユーザーで再度要件整理を行う。
- ループのたびに同じ指摘が繰り返される場合は、[07_decisions.md](./07_decisions.md)に設計判断として記録し、以後の手戻りを防ぐ。

## 9. docs更新ルール

- 実装内容がdocsの記述と食い違う変更を行った場合、**同じタスクの中で該当docsも更新する**（後回しにしない）。対象は主に[03_api_design.md](./03_api_design.md)、[04_data_model.md](./04_data_model.md)、[05_tasks.md](./05_tasks.md)、[06_architecture.md](./06_architecture.md)、[08_screen_design.md](./08_screen_design.md)、[09_deployment.md](./09_deployment.md)。
- 「なぜそう決めたか」という設計判断は[07_decisions.md](./07_decisions.md)に追記する（既存の日付順ADR形式を踏襲する）。
- 現状サマリーである[development_status.md](./development_status.md)は、機能追加・状態変化のたびに更新する（別チャット・将来のAIが最初に読む前提のため、鮮度を保つ）。
- docsと実装の矛盾に気づいた場合（自分が今回変更した箇所以外でも）、黙って直さずユーザーに報告する（[CLAUDE.md](../CLAUDE.md)参照）。

## 10. コミット報告ルール

- コミットメッセージは「何を」ではなく「なぜ」を中心に、簡潔に書く（既存のコミット履歴のスタイルに合わせる）。
- 作業後の報告には、少なくとも以下を含める。
  - 変更したファイルの一覧
  - 実行した検証コマンドとその結果（lint/test/build/pytest）
  - 実際に動作確認した内容（可能な範囲で）
  - 未解決の課題・既知の制約
  - コミットハッシュ（コミットした場合）
- 詳細なフォーマットは [task_template.md](./task_template.md) の「作業後の報告形式」を正とする。
