## 2. 分析画面（トップ、実装済み）

### 目的

ブランド名を入力し、AI上での認知度の推定結果を5つの観点で確認する、本プロダクトの中心画面。

### レイアウト

```
┌───────────────────────────────────────────────────────┐

│ ブランドがAIサービス上でどのように認知されているかを分析します        │
├───────────────────────────────────────────────────────┤
│ ┌───────────────────────────────────────────────────┐ │
│ │ ブランド名 [____________________]      [分析開始]    │ │  ← 入力フォームカード
│ │                                                     │ │
│ │ 分析対象URL（任意・1行に1件・最大10件）                │ │
│ │ [https://example.com/article-a               ]      │ │
│ │ [https://example.com/article-b               ]      │ │  ← 複数行テキストエリア
│ │ [                                             ]      │ │
│ │ 未入力の場合は開発用のサンプル文章で分析します。         │ │
│ │ URL指定時は20〜25秒ほどかかることがあります。           │ │
│ │ （不正なURL・件数超過があればここにエラー表示）           │ │

│ 「〇〇」の分析結果            [共起語のみ実計算...] [リセット] │

│ │  （取得できたページの     │ │                        │   │
```

実装対応: [app/page.tsx](../app/page.tsx)、[BrandInputForm.tsx](../app/components/BrandInputForm.tsx)、[AnalysisDashboard.tsx](../app/components/AnalysisDashboard.tsx)、[CooccurrenceRankingSection.tsx](../app/components/sections/CooccurrenceRankingSection.tsx)、`app/components/sections/*`

### 入力フォームの詳細（ブランド名 + URL）

- ブランド名は必須（従来どおり）。
- URL欄は任意の複数行テキストエリア。1行1件、空行は無視、最大10件（[url-validation.ts](../app/lib/url-validation.ts) の `validateUrlsInput()` が担う）。
- クライアント側で以下を送信前にチェックし、問題があればフォーム内に赤枠でエラー文言を表示して送信をブロックする（API呼び出し自体を行わない）。
  - `http://`/`https://` で始まる妥当なURL形式か（`new URL()` でパース）
  - 件数が10件以内か
  - 重複URLはエラーにせず黙って除外する
- localhost・プライベートIPなど、DNS解決が必要な最終判定はブラウザ側では行わず、引き続きPython側（`services/web_fetcher.py`）が行う。フロントはあくまで「明らかに不正な形式」のみを事前に弾く。
- 送信時、URLが1件もなければ `urls` キー自体をリクエストボディに含めない（`urls: []` は送らない。API側では `urls: []` を明示的な入力エラーとして扱う設計のため）。

### 状態遷移

```
idle（未入力/未実行）
  │ 「分析開始」押下（brandName入力済み、URL欄はバリデーションを通過）
  ▼
loading（ボタン・入力欄を無効化して二重送信を防止）
  │ urls未指定 → 「分析中です。しばらくお待ちください...」
  │ urls指定   → 「Webページを取得・分析しています。20〜25秒ほどかかる場合があります...」
  │
  │ 成功                          │ 失敗（入力エラー400・Python不通等）
  ▼                              ▼
done（5セクション表示 + リセット）   error（エラーメッセージ表示）
  │ 「リセット」押下
  ▼
idle
```

URL欄のバリデーションエラー（不正な形式・件数超過）はAPI呼び出しより前、フォーム内で完結する。API呼び出し自体が失敗した場合（`documents`/`urls` の件数制限超過等、サーバー側検証）は通常の `error` 状態として画面上部にメッセージを表示する。

### 各セクションの表示要素

| セクション | 主な表示要素 |
| --- | --- |
| 1. ブランド認知サマリー | 認知スコア、総言及数、主要プラットフォーム、センチメント内訳（バー）、サマリー文章 |
| 2. 共起語ランキング | 順位、キーワード、出現回数、トレンド（↑↓→）、バー。URL指定時は取得成功/失敗件数のサマリ、全件失敗時は専用メッセージ（下記参照） |
| 3. 文脈分析 | 文脈名、説明、センチメントバッジ、引用例 |
| 4. AI Overview比較 | プラットフォーム名、掲載有無、順位、概要（テーブル形式） |
| 5. 改善提案 | 施策タイトル、説明、優先度バッジ |

### URL取得結果の表示（共起語ランキングセクション内）

- **全件成功**: 「URL取得: 3/3件成功」を共起語ランキングの上に小さく表示。
- **一部成功**: 「URL取得: 2/3件成功（取得できたページのみで分析しています）」と表示し、成功分だけで計算したことを明示。
- **全件失敗**: 「URL取得: 0/3件成功」に加えて、ランキングの代わりに「URLを取得できなかったため共起解析を実行できませんでした」という専用メッセージを表示する（`meta.sections.cooccurrenceRanking: "unavailable"` を根拠に判定。実際に計算して0件だった場合と見分けがつくようにするため）。
- いずれの場合も、URLごとの内部エラー文言（解決済みIPアドレスや例外メッセージ等）はそのまま画面に出さない。件数のみを表示する（[app/lib/meta-label.ts](../app/lib/meta-label.ts) の `getUrlFetchSummary()`）。

### 今後の変更予定

## 3. ブランド一覧画面（未実装）

### 目的

過去に分析したブランドを一覧し、再分析や履歴確認への入口とする。

### レイアウト（案）

```
┌───────────────────────────────────────────────────────┐
│ ブランド一覧                              [+ 新規ブランド] │
├───────────────────────────────────────────────────────┤
│ ブランド名     最終分析日        認知スコア    操作         │
│ OpenAI        2026-07-09        78          [再分析][履歴]│
│ Anthropic     2026-07-08        —           [再分析][履歴]│
└───────────────────────────────────────────────────────┘
```

- 「新規ブランド」押下でモーダル or 簡易フォームを表示し、`POST /api/brand` を呼ぶ。
- 「再分析」押下でトップ画面（分析画面）に遷移し、当該ブランド名を入力済みの状態で開始できるようにする。
- 「履歴」押下で分析履歴画面（4章）に遷移する。
- 使用API: `GET /api/brand`, `POST /api/brand`

## 4. 分析履歴画面（未実装）

### 目的

過去の分析結果を振り返り、必要であれば「この結果はどこから来たか」を確認できるようにする。

### レイアウト（案）— 一覧

```
┌───────────────────────────────────────────────────────┐
│ 分析履歴                        ブランド: [OpenAI ▾]      │
├───────────────────────────────────────────────────────┤
│ 実行日時            ステータス   認知スコア                │
│ 2026-07-09 10:00   完了        78            [詳細を見る] │
│ 2026-07-01 09:30   完了        72            [詳細を見る] │
└───────────────────────────────────────────────────────┘
```

### レイアウト（案）— 詳細（`/history/:analysisId`）

トップ画面の分析結果（2章のセクション1〜5）と同じダッシュボードを再利用して表示する。加えて、各項目の横に情報源バッジを表示する:

```
┌───────────────────────────────────────────────────────┐
│ 2. 共起語ランキング                                       │
│ 1. 料金プラン   42件  ↑   [情報源: PR TIMES, News]        │
│ 2. 導入事例     35件  ↑   [情報源: Common Crawl]          │
└───────────────────────────────────────────────────────┘
```

- 情報源バッジはクリックで参照元URL一覧を表示（ポップオーバー等）。
- 使用API: `GET /api/history`, `GET /api/history/:analysisId`
- データ根拠: [04_data_model.md](./04_data_model.md) の `analysis_sources` / `analysis_result_sources`

## 7. デザインの共通方針

- Tailwind CSSによる管理画面風レイアウト（`zinc`系グレースケール＋アクセントカラーはセンチメント/トレンド/優先度バッジのみ）
- ライト/ダークモード両対応（`dark:` バリアント）
- カードベースのレイアウト（[Card.tsx](../app/components/Card.tsx) を共通コンポーネントとして再利用）
- 新規画面を追加する際も、既存の `Card` コンポーネントとバッジスタイル（[badge-styles.ts](../app/lib/badge-styles.ts)）を再利用し、見た目の一貫性を保つ
